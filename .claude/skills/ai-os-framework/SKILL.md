# AI OS Framework — 个人 AI 工作系统框架

> **使用方式说明：** 这不是一个"调用后执行某个任务"的操作型 Skill，而是一个元框架——定义 agent 怎么记忆、怎么协作、怎么沉淀知识。加载本 Skill 时会自动执行安装检查，无需手动配置。

## 首次加载 — 自动安装

当本 Skill 被调用时，在执行任何其他操作之前，先完成以下自检：

1. 读取项目根目录的 `CLAUDE.md`（如不存在则创建）
2. 搜索其中是否包含 `AI OS Framework 核心约束` 这个标题
3. 如果**不存在**，将以下内容追加到 `CLAUDE.md` 末尾：

```markdown
## AI OS Framework 核心约束

来自 ai-os-framework v1.10.0，每会话自动生效。全文见 /ai-os-framework。

### 行为准则
1. 先检索 memory/ 再回答，不凭记忆推测
2. 不确定就标"待确认"（确认/待确认/推测 三级置信度）
3. 会话启动时主动读 MEMORY.md 和项目文档，反馈待办

### 做即记
以下事件发生时当场写入项目文档，不堆积：
- 做出决策 → 决策记录（日期 + [分类标签] + 决策 + 为什么）
- 踩坑解决 → 技术坑点（问题→根因→解法）
- 模块完成 → 更新模块清单状态（不确定是否算完成时主动问用户确认）
- 新需求 → 待办与优先级

### 节奏锚点
每完成一个独立任务（写完一个功能、修完一个 bug、给出一组建议），自检一次：刚才有没有需要写入项目文档的事？有就记，没有就过。不在会话结束时才检查，在每个任务节点都触发。

### 日志回查
会话启动时检查最近 3 天日志，搜索与本项目相关的判断偏差和交接笔记。
```

4. 如果**已存在**但版本号不一致，将版本号更新为 v1.9.0，其余内容不变
5. 告知用户：安装完成 / 已是最新，然后继续执行用户指令

此后每次新会话，agent 会自动加载 CLAUDE.md 中的核心约束。需要查阅完整协议时再调用本 Skill。

## 一、必读

以下原则适用于所有 agent、所有项目，没有例外。

### 核心命题

不要只使用 AI，要构建一套会执行、会记忆、会生长的个人 AI 工作系统。

"使用 AI"每次只得到一个输出。"构建系统"在积累一套能力。短期看差别不大，长期看差距是指数级的。

### 三个差距

AI 不替代人，但会和 AI 协同工作的人，会替代不会的人。差距来自三个维度：

- **效率差**：信息整理、初稿生成、方案推演、结果审查，协同者快一个数量级
- **判断差**：AI 帮你看全信息、风险、假设和反例，判断质量更高
- **复利差**：每次协同的经验进入长期记忆，下一次不再从零开始

### 输出质量公式

```
AI 输出质量 = 模型能力 × 上下文质量 × 人类判断 × 反馈沉淀
```

四个因子缺一不可。同一个模型，有人惊艳有人平庸，差别在输入、判断和沉淀，不在工具本身。

垃圾进，垃圾出：问题模糊、背景缺失、目标不清、没有复盘，AI 产出的一定是"看起来完整但不可执行"的东西。

### 三层架构

```
人类判断层 — 定义方向、约束、取舍、责任
AI 执行层 — 分析、推演、生成、推进任务到完成状态
知识记忆层 — 记录、连接、复盘，让每次判断更快更准
```

人最重要。没有人的目标和判断标准，AI 输出再多也只是更快地产生噪音。

### 协作闭环

```
感知 → 决策 → 行动 → 反馈 → 沉淀
```

不是单次问答，是闭环。输出有没有被采用？执行中哪些假设是错的？经验有没有回到知识库？闭环断了，飞轮就不转。

### 积累原则

记忆层的价值不是"记更多"，是"让下一次判断更快更准"。

值得沉淀的：反复用到的判断标准、踩过的坑、可复用的方法、决策依据、能让 AI 下次更好理解你的上下文。

不值得沉淀的：临时信息、一次性碎片、只用一次的调试过程。

知识库不是仓库，是参考手册。越大越难用，越精越好用。

### 框架边界

本框架定义的是知识管理和协作协议层——解决的是"信息怎么记、经验怎么沉淀、agent 怎么协作"的问题。它不替代具体的技术选型、产品设计和业务判断。方向对不对、方案好不好、风险大不大，这些永远是判断层的职责，框架只负责让你的判断越来越有据可依。
