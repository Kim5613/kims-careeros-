# 项目知识库 — 操作手册

本文档是 AI OS Framework 的详细操作手册。搭建新项目记忆系统时必须完整阅读。

---

## 三层角色定义

| 层 | 职责 | 谁承担 |
|---|---|---|
| 判断层 | 目标、取舍、责任、验收标准 | 人 |
| 执行层 | 把模糊问题推进成结构化结果 | agent |
| 记忆层 | 记录、检索、让经验进入下次上下文 | MEMORY.md + 项目文档 + Skill |

---

## 知识积累规则

| 事件类型 | 是否记录 | 沉淀位置 |
|---|---|---|
| 技术选型 / 方案选择 | ✅ 必须 | 项目文档 → 决策记录 |
| 踩坑并解决 | ✅ 必须 | 项目文档 → 技术坑点；有通用价值 → Skill |
| 模块完成 | ✅ 必须 | 项目文档 → 模块清单 |
| 新需求 / 需求变更 | ✅ 必须 | 项目文档 → 待办与优先级 |
| 日常调试 / 小修复 | ❌ 不记 | — |
| 临时讨论 / 探索 | ❌ 不记 | — |
| 可复用操作流程 | ✅ 必须 | Skill（创建时调用平台 Skill `create-skill` 获取格式标准） |

判断标准：下次遇到同样的事，这条信息能否让我少走弯路？能，就记；不能，就不记。

### 知识提炼层次

记录不等于搬运原文。每条知识应按以下层次提炼，尽量往高层走：

| 层次 | 含义 | 示例 |
|---|---|---|
| Source | 原始事实：发生了什么 | "6月22日 antd 样式丢失" |
| Concept | 提炼方法：说明了什么规律 | "cssinjs 多版本实例会导致 React Context 不互通" |
| Synthesis | 交叉判断：和其他经验结合形成什么通用原则 | "SSR 项目中 CSS-in-JS 库必须保证单实例，通过 overrides 锁定版本" |

Source 级别的信息可以记在日志里。进入项目文档和 Skill 的内容，至少要达到 Concept 级别。反复出现的 Concept 应主动合并为 Synthesis。

---

## 日志格式

memory/YYYY-MM-DD.md 的标准格式：

```markdown
## [项目名] 事件类型
日期 | 关键内容摘要 | 沉淀位置（项目文档/Skill）
```

示例：

```markdown
## [CareerOS] 决策
2026-06-21 | 文档识别全部本地处理 | 项目文档 → 决策记录

## [CareerOS] 问题解决
2026-06-22 | antd CSS-in-JS 版本冲突导致 SSR 样式丢失 | Skill: antd-cssinjs-ssr-fix

## [CareerOS] 阶段完成
2026-06-21 | 文档识别与简历导入功能完成，11 步全部落地 | 项目文档 → 模块清单

## [巡检] 无异常
2026-06-23 | 巡检完成，未发现缺漏
```

巡检任务每次执行后，无论是否发现缺漏，都必须写入一条巡检日志。无异常也要记录，确保巡检可追溯。

### 判断偏差日志（协作日志）

当事件日志记录的是"发生了什么"（Source 级），判断偏差日志记录的是"我为什么判错了"（Synthesis 级）。当用户纠正了你的判断、或事后发现决策有误时写入：

```markdown
## [项目名] 判断偏差
日期 | 任务：做了什么 | 我的判断：当时怎么想的 | 实际偏差：哪里错了 | 下次怎么改
```

示例：

```markdown
## [ai-os-framework] 判断偏差
2026-06-24 | 任务：SKILL.md v1.4.0 结构设计 | 我的判断：理论前置（核心命题→三层架构）、行动后置（模板→搭建步骤），符合"先理解再动手" | 实际偏差：接收方读完理论不知道第一步该做什么，反馈"看不懂怎么用" | 下次怎么改：设计文档入口时先给行动路径再给理论，"先做后补"比"先懂后做"更适合操作型文档
```

判断偏差日志不需要每次都写。只在"下次遇到类似情况，这条记录能帮我少走弯路"时才记。

## 协作协议

### 会话启动协议

1. 读 MEMORY.md → 了解活跃项目和当前状态
2. 主动检索相关项目文档（memory/project-xxx.md）和记忆 → 拿到事实依据
3. 检索待办事项 → 主动反馈：当前有哪些未完成工作、优先级如何
4. 搜索最近 3 天日志 → 检查有无与本项目相关的未闭合事项（未解决的坑、待确认的决策、未完成的交接笔记）
5. 根据用户提问判断本次涉及哪个项目，补充读取对应文档
6. 开始工作

核心原则：先检索，再开口。不得凭记忆推测。

### 会话结束协议

**主机制：做即记。** 不需要等对话结束。以下事件发生时当场写入：

- 做出决策 → 当场追加到项目文档决策记录
- 踩坑找到解法 → 当场写入技术坑点
- 模块开发完成 → 当场更新模块清单状态（不确定是否算完成时主动问用户确认）
- 收到新需求 → 当场写入待办

**兜底：收尾扫描。** 检测到以下信号时快速扫一眼：

- 用户说"先这样""今天就到这""下次再聊""bye"
- 连续两轮无新任务
- 用户切换到另一个项目

有漏就补，没漏就过。不强制写结束日志。

### 节奏锚点

做即记依赖 agent 在任务节点主动触发，而不是等会话结束。为对抗长对话中的约束衰减，每完成一个独立任务（写完一个功能、修完一个 bug、给出一组建议），自检一次：

> 刚才有没有需要写入项目文档的事？

有就记，没有就过。频率越高，衰减越来不及发生。

### 自动巡检协议（每 24 小时）

1. 过去 24 小时的会话中，有没有该记没记的决策、踩坑、模块完成
2. 项目文档的模块清单和待办是否和实际进度一致
3. 有没有反复出现的操作还没沉淀为 Skill
4. MEMORY.md 的指针是否还准确，有没有过时信息
5. 如有缺漏，自动补齐到对应文档
6. 无异常时也要写入巡检日志（`## [巡检] 无异常`），确保巡检可追溯——没日志 = 不知道跑了没

**cron 配置参考：**

```
schedule: "0 9 * * *"  (每天 9:00)
timezone: 按用户所在地设置

prompt 模板：
执行每日巡检协议。检查过去 24 小时的会话记录：
1. 有没有该记没记的决策、踩坑、模块完成
2. 项目文档的模块清单和待办是否和实际进度一致
3. 有没有反复出现的操作还没沉淀为 Skill
4. MEMORY.md 的指针是否还准确
如有缺漏，补齐到对应文档。完成后写入巡检日志到 memory/YYYY-MM-DD.md。
```

### 周期复盘协议（每周）

巡检做的是查漏补缺，复盘做的是提炼升华。每周执行一次：

1. 回顾本周的日志和项目文档，找出反复出现的模式
2. 把零散的技术坑点归纳为方法论（从 Concept 升级到 Synthesis）
3. 把多次决策记录提炼为判断标准（"遇到这类问题时，我们的原则是……"）
4. 检查项目文档是否需要结构性调整（章节拆分、合并、重排优先级）
5. 将复盘结论写入项目文档的决策记录或技术坑点，并写入复盘日志

复盘日志格式：

```markdown
## [项目名] 周复盘
日期 | 本周关键模式 | 提炼的判断标准或方法论
```

复盘和巡检的区别：巡检问"有没有漏记"，复盘问"这些记录合在一起说明了什么"。

### 上下文切换协议

当同一个 agent 或用户在多个项目之间切换时，防止上下文丢失和交叉污染。

**切换前（保存状态）：**

在当前项目的日志中写入一条交接笔记，确保下次回来能无缝续接：

```markdown
## [项目名] 交接
日期 | 正在做：当前任务 | 卡在哪：阻塞点或待确认项 | 下一步：回来后的第一个动作
```

示例：

```markdown
## [BookMark] 交接
2026-06-24 | 正在做：标签系统 API | 卡在哪：多标签筛选的性能未验证 | 下一步：跑 benchmark，确认查询耗时
```

交接笔记不需要写完整的工作总结——那是会话结束协议的事。交接笔记只回答一个问题："下次回来时，我需要知道什么才能立刻继续？"

**切换后（加载上下文）：**

按会话启动协议执行，额外检查上一次的交接笔记。如果交接笔记中的"卡在哪"已经解决，更新状态。

**多 agent 并发规则：**

当多个 agent 同时服务于不同项目时：

- 每个 agent 只加载自己负责项目的上下文，不主动读取其他项目文档
- 回复时严格使用当前项目的上下文，不把 A 项目的技术决策、架构约定混入 B 项目的回答
- 如果用户提到了另一个项目的信息且与当前任务相关，显式标注"这来自 [项目名] 的上下文，是否要在当前项目中参考？"而不是默默混入

当多个 agent 同时服务于同一个项目时，遵循并发写入规则（见上文）。

### agent 行为准则

以下五条是 agent 的硬性约束，适用于所有项目、所有会话：

1. **先检索，再回答**：所有关于项目状态、技术细节、历史决策的回答，必须先检索确认，不凭记忆推测
2. **不确定就说不确定**：对事实没把握时，按置信度标记规则显式标注，不默默猜测
3. **做即记**：决策、踩坑、模块完成、新需求——发生时当场写入，不堆积到会话结束
4. **主动反馈待办**：会话启动时主动检索并汇报未完成事项
5. **输出后自检**：生成重要内容（方案、报告、架构设计）后，主动站在审查者角度反向检查——是否符合目标、有无遗漏、逻辑是否自洽、哪些假设未经验证

## 置信度标记

对事实没把握时，不要默默猜测，显式标记：

- **确认**：从项目文档或代码中检索到的事实，直接引用来源
- **待确认**：有线索但不确定，说明依据并请求用户确认
- **推测**：完全没有依据的推断，明确标注"这是我的推测，需要验证"

示例：

> "根据项目文档记录，该模块状态为开发中。但本次会话中你提到已经改完了，需要你确认是否更新为已完成。"

## 文件模板

### MEMORY.md 模板（控制在 3KB 以内）

```markdown
## 活跃项目
[项目名] 一句话描述 | 技术栈 | 当前状态
→ 项目文档: memory/project-xxx.md

## 跨项目经验（详见 Skills）
- 经验描述 → 指向 Skill 或项目文档

## 沉淀规则
- 只记可复用的判断，不记流水账
- 项目细节落项目文档，这里只放指针
- 踩坑经验提炼为 Skill，不散落在日志里
- 会话开始前先检索上下文，不凭记忆推测
```

**超 3KB 时的拆迁规则：**

MEMORY.md 只做路由表，不做知识库。接近或超过 3KB 时按以下优先级拆迁：

- 某个项目的描述超过一行 → 细节移入 `memory/project-xxx.md`，MEMORY.md 只保留一行索引
- 跨项目经验积累到 3 条以上且主题相近 → 提炼为一个 Skill，MEMORY.md 只保留 Skill 名称指针
- 已完成超过 2 周的项目 → 从活跃项目区移除，仅在项目文档中保留归档状态
- 任何可以从文件名或目录结构推断的信息 → 删除，不占 MEMORY.md 空间

拆迁时机：巡检协议第 4 条（"MEMORY.md 的指针是否还准确"）是天然的检查点。每次巡检时发现超限就执行拆迁。

### memory/project-xxx.md 模板

```markdown
# [项目名]

## 一句话定位
这个项目是什么、为谁做的、解决什么问题

## 当前状态
做到哪了、接下来做什么、有没有卡住的地方

## 架构概览
技术栈、关键目录结构、模块之间的依赖关系

## 模块清单
| 模块 | 状态 | 关键说明 |
|---|---|---|
| ... | ✅ / 🔧 / ⏳ | ... |

## 决策记录
按时间倒序。每条格式：日期 + [标签] + 决策 + 为什么
标签分类建议：[技术选型] [架构] [流程] [工具] [需求] [隐私]
用途：决策积累到 10 条以上时，按标签检索比按时间扫描快得多

## 技术坑点
每条：问题 → 根因 → 解法（可独立阅读）

## 待办与优先级
按优先级排序的需求列表

## 近期关键事件
<!-- 最近 3 条，旧事件溢出后删除（日志里有完整版） -->
- [日期] [类型] 摘要 → memory/YYYY-MM-DD.md
```

### memory/YYYY-MM-DD.md 日志模板

```markdown
## [项目名] 事件类型
日期 | 关键内容摘要 | 沉淀位置（项目文档/Skill）
```

## 并发写入规则

当多个 agent 同时操作同一个项目时：

- 每个 agent 只更新自己负责的模块段落
- 写入前先读取项目文档最新版本，检查是否有冲突
- 如果发现其他 agent 刚更新了同一区域，标记"待确认"而不是直接覆盖
- 日志文件追加写入（不覆盖已有条目）

## 冷启动最小集

新项目刚立项时，项目文档只需填四个字段即可启动：

```markdown
# [项目名]

## 一句话定位
[一句话说清楚这是什么项目]

## 当前状态
刚立项，尚未开始开发

## 架构概览
技术栈：[已确定的技术栈]

## 待办与优先级
1. [第一个需求]
```

架构、模块清单、决策记录、技术坑点，都留到实际工作中边做边补。冷启动的门槛要足够低，否则"搭建记忆系统"本身就变成了第一个阻力。

前几次会话的重点是：做出第一个决策、踩到第一个坑、完成第一个模块——这些事件自然触发沉淀规则，知识库就开始生长了。

---

## 从零搭建步骤

1. 创建 MEMORY.md → 按模板写入项目索引
2. 创建 memory/project-xxx.md → 按冷启动最小集填写基本信息
3. 更新 AGENTS.md → 写入会话启动协议、会话结束协议、沉淀规则、巡检协议
4. 提取已有的踩坑经验 → 创建对应的 Skill
5. 创建定时巡检任务 → 每天固定时间执行巡检协议（cron，云端运行）
6. 验证 → 开一个新会话，确认 agent 能按启动协议正确加载上下文，并能主动反馈待办

## 框架自身的记忆管理

本框架也是一个项目——它的迭代决策、结构变更、版本取舍同样需要被记录。但框架的记忆文件不属于分发包（zip 是给别人的，记忆是自己的）。

**实践方式：** 在你的工作目录下，用本框架的模板建 `memory/project-ai-os-framework.md`，像管理其他项目一样管理框架自身的迭代。每次结构性变更后记一条决策，格式和其他项目文档一致。

SKILL.md 的 version 字段是框架的唯一版本号来源，每次发布必须更新。
