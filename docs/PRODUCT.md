## 一、产品定位

### 一句话

**HR 从业者的个人职业操作系统 — 自己的职业生涯，用管理产品的思路来管。**

### 核心价值

| 维度 | 痛点 | 解法 |
|------|------|------|
| 求职管理 | 投了哪些、面到哪轮、薪资谈到哪步，全靠脑子记 | 结构化追踪求职全流程 |
| 职业资产 | 简历散落各处、面试话术每次都重写 | 简历版本库 + 话术库 + 求职信模板 |
| HR 工作 | 候选人信息零散、面试评价无沉淀 | 候选人库 + 面试记录 + 简历解析 |
| 知识沉淀 | 面试标准、岗位评估、薪酬数据每次重新查 | 招聘知识库 + 市场洞察 |
| 成长复盘 | 不知道这一年做了什么、涨薪靠感觉 | 成就记录 + 晋升时间线 + OKR 追踪 |

### 不是什么

- **不是 ATS（招聘管理系统）** — 不对外、不协同、不上线给第三方用
- **不是 HR SaaS** — 不需要多租户、不需要权限体系
- **不是通用工具** — 只服务 Kim 一个人的场景，不做泛化

## 二、用户画像

**唯一用户：Kim（HR 从业者）**

核心工作流：
1. **管理自己的职业生涯** — 跳槽、涨薪、晋升、技能成长
2. **做 HR 招聘工作** — 筛选候选人、安排面试、沉淀招聘知识
3. **建立长期职业资产** — 简历迭代、话术积累、人脉管理

**技术交付物**：
- 14 个 Prisma 数据模型（schema.prisma）
- 19 个 API 路由（全 Prisma 实现，但只支持 GET 列表 + POST 创建）
- 文档解析引擎：PDF（pdfjs-dist）+ 图片 OCR（tesseract.js 中文）+ Markdown/文本
- FileUpload 组件（拖拽上传 + 进度条 + 格式校验）
- AttachmentList 组件（下载/解析预览/删除）
- 文件上传/下载/解析/附件管理 4 个 API
- 阿里云部署脚本（deploy.sh）+ Nginx 配置

**未完成**：
- 无数据库实例（Prisma schema 未执行）
- 页面数据全为 Mock，刷新即丢失
- 页面间数据不互通（候选人库新增的人在首页看不到）
- 缺编辑/删除 API（无 PATCH/DELETE 路由）

## 九、v1.2 规划 — 人生设计 × CareerOS 集成

> 来源：2026-07-08 个人人生设计蓝图讨论。Kim 的正北方向是 **"用一种不需要任何人批准的方式，做出能被别人看见的东西"**。CareerOS 不只是管理工具，也是这条路上的导航仪。

### 9.1 能量日记

**场景**：蓝图里的"能量日记实验"——每天标注哪件事回血、哪件事抽干。

**落地方向**：首页日视图的每个待办/日程完成后，一键标记"⚡回血"或"🪫消耗"。一周后自动汇总能量报告。

### 9.2 创作项目

**场景**：方案二的 AI 漫剧、建网站、学缝纫/长笛/萨克斯——这些不是"工作成就"，但它们是 Kim 真正在生长的部分。

**落地方向**：成就库新增 `type` 字段——工作成就 / 创作项目 / "第一次"。能力地图的技能可以同时属于方案一（HR路径）和方案二（创作者路径）。

### 9.3 路径标签（奥德赛计划分组）

**场景**：蓝图里的三个奥德赛计划——方案一（稳中求变）、方案二（AI创作者）、方案三（自由感官人生）。每个能力/成就/里程碑可以标注"服务于哪个路径"。

**落地方向**：能力地图、成就库、里程碑统一加 `odyssey` 标签字段。能力雷达按路径分组对比，看到底"哪个奥德赛在真实生长"。

### 9.4 "第一次"里程碑

**场景**：蓝图里说的"种一棵树"——建网站是第一次、部署到云端是第一次、买域名是第一次。这些"第一次"是 Kim 人生中最有生命力的节点。

**落地方向**：里程碑新增 `type: 'first-time'` 类型，卡片样式与普通里程碑区分（如彩色边框、特殊角标）。

### 9.5 首页能量小组件

**场景**：蓝图里"守卫睡眠"实验、打断自动驾驶的随机提醒。

**落地方向**：首页日视图顶部或侧边添加小型能量仪表盘——今日回血/消耗比例、连续睡眠底线天数、一条随机的人生设计提醒。
