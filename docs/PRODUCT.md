# Kim's CareerOS — 产品文档

> 本文档定义产品愿景、用户场景、功能边界和需求优先级。所有新需求先对照本文档判断是否属于当前范围，再决定做不做、排哪期。

---

## 一、产品定位

### 一句话

**HR 从业者的个人职业操作系统 — 自己的职业生涯，用管理产品的思路来管。**

### 核心价值

| 维度 | 痛点 | 解法 |
|------|------|------|
| 求职管理 | 投了哪些、面到哪轮、薪资谈到哪步，全靠脑子记 | 结构化追踪求职全流程 |
| 职业资产 | 简历散落各处、面试话术每次都重写 | 简历版本库 + 话术库 + 求职信模板 |
| HR 工作 | 候选人信息零散、面试评价无沉淀 | 候选人库 + 面试记录 + 简历解析 |
| 知识沉淀 | 面试标准、岗位评估、薪酬数据每次重新查 | 招聘知识库 + 市场洞察 |
| 成长复盘 | 不知道这一年做了什么、涨薪靠感觉 | 成就记录 + 晋升时间线 + OKR 追踪 |

### 不是什么

- **不是 ATS（招聘管理系统）** — 不对外、不协同、不上线给第三方用
- **不是 HR SaaS** — 不需要多租户、不需要权限体系
- **不是通用工具** — 只服务 Kim 一个人的场景，不做泛化

---

## 二、用户画像

**唯一用户：Kim（HR 从业者）**

核心工作流：
1. **管理自己的职业生涯** — 跳槽、涨薪、晋升、技能成长
2. **做 HR 招聘工作** — 筛选候选人、安排面试、沉淀招聘知识
3. **建立长期职业资产** — 简历迭代、话术积累、人脉管理

## 四、版本历史

### v0.1 — 可交互原型（2026-06-21，qoder）

**做了什么**：10 个页面全部有 UI，所有数据为前端 Mock，API 层已写 Prisma 但页面未调用。

| 模块 | 实现内容 |
|------|----------|
| 首页看板 | 快捷操作卡片、静态动态时间线、待办日历、模块导航 |
| 求职管理 | 看板/表格双视图、投递记录 CRUD、状态筛选、来源筛选、日期范围筛选 |
| 薪酬与晋升 | 晋升时间线、薪资变化列表、新增表单 |
| 成长档案 | 里程碑列表、成就列表、OKR 目标看板（进行中/已完成/已放弃）、年度复盘 |
| 简历与求职信 | 简历版本管理（三栏卡片）、求职信模板、自我介绍话术库（按场景分组） |
| 候选人库 | 候选人表格、状态标签（可点击切换）、人才库标记、详情抽屉、面试时间线 |
| 招聘知识库 | 岗位知识卡片（按分类筛选）、新增/编辑表单、搜索 |
| 市场洞察 | 洞察卡片列表、分类/行业筛选、新增表单 |
| 公司库 | 公司表格、行业/城市筛选、详情抽屉（关联投递数/候选人数） |
| 人脉库 | 联系人卡片/表格双视图、关系类型筛选、搜索 |

**技术交付物**：
- 14 个 Prisma 数据模型（schema.prisma）
- 19 个 API 路由（全 Prisma 实现，但只支持 GET 列表 + POST 创建）
- 文档解析引擎：PDF（pdfjs-dist）+ 图片 OCR（tesseract.js 中文）+ Markdown/文本
- FileUpload 组件（拖拽上传 + 进度条 + 格式校验）
- AttachmentList 组件（下载/解析预览/删除）
- 文件上传/下载/解析/附件管理 4 个 API
- 阿里云部署脚本（deploy.sh）+ Nginx 配置

**未完成**：
- 无数据库实例（Prisma schema 未执行）
- 页面数据全为 Mock，刷新即丢失
- 页面间数据不互通（候选人库新增的人在首页看不到）
- 缺编辑/删除 API（无 PATCH/DELETE 路由）

### v0.2 — MVP（2026-06-24）

**做了什么**：接入数据库 + 补全 API + 核心页面真实数据驱动 + 跨页面动态聚合。系统可以日常使用了。

**后端**

| 交付物 | 说明 |
|--------|------|
| PostgreSQL 17 数据库 | 本地安装，21 张表已建，scram-sha-256 认证 |
| 15 个 [id] 动态路由 | 全部资源支持 PATCH（编辑）和 DELETE（删除），含公司关联处理、名称冲突检测 |
| 首页动态聚合 API | `/api/dashboard/activity`：从 8 个业务表拉取最近记录，统一格式 + 相对时间展示 |
| CoverLetter 关系修复 | schema 中补了 `@relation` 和反向字段 |
| UPLOAD_DIR 适配 Windows | 从 `/data/careeros-uploads` 改为 `D:/data/careeros-uploads` |

**前端 — 共享层**

| 交付物 | 说明 |
|--------|------|
| `useApiList` hook | 通用 API 数据管理：自动 fetch → API-first 读写 → API 不可用时降级 Mock |

**前端 — 页面迁移**

| 页面 | 迁移内容 |
|------|----------|
| 候选人库 | Mock → API 驱动。新增/编辑/删除/状态变更/人才库标记全部走 API。面试记录走 `/api/candidate-interviews`。上线文同步（数据更新后详情抽屉自动刷新） |
| 简历与求职信 | 3 个实体的 CRUD 全部走 API（resumeCreate/Update/Remove），删除/设默认/复制均调用 API，去掉"(本地)"降级提示 |
| 求职管理 | Mock → API 驱动。新增/编辑/删除走 API。看板/表格双视图数据来自数据库 |
| 公司库 | API 驱动。新增/编辑走 API（含名称冲突检测），删除带确认弹窗 |
| 人脉库 | API 驱动。新增/编辑走 API（含公司关联），删除带确认弹窗 |
| 招聘知识库 | Mock → API 驱动。新增/编辑/删除走 API |
| 市场洞察 | Mock → API 驱动。新增走 API |
| 首页看板 | 动态时间线改为从 `/api/dashboard/activity` 拉取真实数据，Mock 为降级后备 |

**未完成**

| 模块 | 说明 |
|------|------|
| 薪酬与晋升 | 仍用 Mock 数据，待下版本 API 迁移 |
| 成长档案 | 仍用 Mock 数据，待下版本 API 迁移 |
| 用户认证 | 单用户场景暂缓，部署前必须加 |

## 五、需求优先级框架

新需求按以下四象限判定优先级：

```
          紧急
            │
  Ⅱ 排下期  │  Ⅰ 现在就做
            │
 不重要 ────┼──── 重要
            │
  Ⅳ 不做   │  Ⅲ 规划中
            │
          不紧急
```

## 六、需求提交流程

为避免"只提需求很混乱"，后续需求按这个格式来：

```
【需求】一句话描述

### 场景
什么时候、做什么事时，遇到什么问题

### 期望
希望系统帮我做什么（不是怎么做，是做什么）

### 优先级
现在就要 / 下个迭代 / 以后再说
```

收到需求后，我会对照本文档判断：
1. 是否在当前产品范围内？
2. 和已有功能有没有冲突？
3. 应该排在哪个优先级？

然后把决策写入 `memory/project-kims-careeros.md` 的决策记录。
